Общая идея

При работе со списками все начинается с директивы rtList. Она представляет собой root container для dependency injection и конфигурирует зависимости для вложенных компонентов (paging-контролы, директивы сортировки и т.д.).

В качестве необходимого минимума для данной директивы необходимо указать метод, который список будет вызывать для получения данных. Данный метод должен принимать один параметр, в который будут переданы параметры запроса. Передаваемые параметры следуют контракту ListRequest для простых списков и PagedListRequest для paged-списков.

Возвращать данный метод должен Observable, Promise либо EventEmitter, прдставляющий для простых списков простой массив элементов. Для paged-списков ответ должен следовать контракту ListResponse

Как подключить

Функционал списков поставляется в модуле RTListsModule Также можно импортировать модуль RtModule, включающий в себя функционал всей библиотеки целиком.

Свойства директивы rtList

Параметр loadOnInit

Данный параметр указывает списку, что необходимо автоматически загрузить данные после того, как список проинициализирован. По умолчанию равен true

Событие onListInit

Данное событие позволяет встроиться в конвеер инициализации списка и выполнить дополнительную настройку перед первой загрузкой данных. Например, можно зарегистрировать объект с настройками фильтра при помощи вызова метода RtList.registerFilterTarget, передаваемого в качестве параметра сервиса RtList (если ваш фильтр представляет собой отдельный компонент, то гораздо проще использовать директиву rtRegisterAsFilter, регистрируя фильтр прямо в шаблоне).

Данное событие генерируется после того, как сработыет хук AfterViewInit и выполнена первая отрисовка шаблона.

Событие afterListInit

Данное событие вызывается после того, как была выполнена инициализация списка и он готов к использованию.

Данное событие, как и предыдущее, передает в качестве параметра объекти типа RtList, представляющий собой сервис для работы со списком.

Вспомогательные методы и свойства директивы rtList

В директиве rtList содержится ряд полезных методов и свойств, которые позволяют легко реализовать типичные сценарии (к примеру, кнопку перезагрузки данных). Ниже приведен список методов и свойств:

  • Метод loadData - выполняет загрузку данных в список.
  • Метод reloadData - выполняет перезагрузку (загрузку со сбросом параметров пейджинга) данных в список.
  • Метод cancelRequests - выполняет отмену выполняющегося в данный момент времени запроса данных.
  • Метод resetSettings - выполняет сброс параметров списка (сортировки, пейджинг, фильтры).
  • Свойство busy - возвращает true если в данный момент времени выполняется запрос за данными.
  • Свойство ready - возвращает true если в данный момент времени запрос за данными НЕ выполняется.

Ниже приведен пример списка с четырьмя кнопками, использующими описанные выше методы и свойства.

Параметр defaultSortings директивы rtList

Данный параметр позволяет задать для списка сортировки по умолчанию. Сортировки будут применены сразу при создании списка и в них же будет сбрасываться значение сортировок при вызове метода resetSettings.

В качестве значения данный параметр принимает массив объектов, удовлетворяющих контракту SortParameter

Выставлять сортировки по умолчанию можно любой момент жизни списка. Если в момент выставления сортировок по умолчанию список еще не был отсортирован, то данные сортировки применятся автоматически.

Ниже приведен пример списка, у которого по умолчанию выставлены сортировки по названию аэропорта в прямом порядке и стране нахождения в обратном, а так же код контракта SortParameter.

Директива rtSort

Данная директива предназначена для настройки сортировок списка. Директива принимает в качестве обязательного параметра идентификатор сортировки, который будет передаваться в запросе за данными.

Ниже расположен живой пример списка с сортировкам, а так же контракт AirportsListRequest. Моменты, на которые следует обратить внимание:

  • К элементу, помеченному данной директивой, добавляется класс rt-sortable. С помощью данного класса вы можете настроить стили, сигнализирующие, что элемент сортируемый. Например, в наших примерах это появление двунаправленной стрелки справа от элемента при наведении мыши.
  • При клике по элементу переданный параметр-идентификатор добавляется в коллекцию примененных сортировок с направлением Asc и тут же вызывается метод загрузки данных, в который передается массив sortings с указанной сортировкой в указанном направлении. К самому элементу добавляется класс rt-sort-asc, с помощью которого можно настилить внешний вид отсортированного столбца.
  • При повторном клике направление сортировки меняется на противоположное и выполняется запрос за данными. Класс rt-sort-asc элемента изменяется на rt-sort-desc.
  • С помощью клика с зажатой клавишей Ctrl можно настроить сортировку по нескольким полям. В таком случае, в массив sortings будет передаваться несколько значений.

Если по какой-либо причине вас не устраивают названия классов rt-sortable, rt-sort-asc b rt-sort-desc, вы можете поменять их глобально. Для этого импортиру те в вашем AppModule сервис SortingsService, и настройте название при помощи статического объекта settings.

rtRowNumber pipe

Данный pipe может быть использован в шаблонах элемента списка для автоматической нумерации строк. Он распознает с каким пейджером работает и проставляет соответствующим образом номера, например, начиная нумерацию не с 1, а с 6 для второй страницы постраничного списка.

Данный pipe применяется к значению index, представляющему собой индекс текущего элемента в коллекции. Также pipe-у необходимо передать директиву rtList в качестве параметра.
